😰 Problem: Bez Generics-a (pre-Java 5)

Scenario: Lista igrača u gaming lobby-ju

❌ Prije Java 5 - Sve je Object
// Lista može sadržavati bilo što!
List players = new ArrayList();

// Dodajemo igrace
players.add(new Player("xXDragonSlayerXx"));
players.add(new Player("NoobMaster69"));

// Netko slučajno doda String umjesto Player objekta!
players.add("Ovo nije Player!");  // 😱 Kompajler to dozvoli!

// Izvlačenje podataka - OPASNO!
for (int i = 0; i < players.size(); i++) {
    Player p = (Player) players.get(i);  // ⚠️ Casting obavezan
    System.out.println(p.getUsername());
}

// 💥 RUNTIME ERROR: ClassCastException!
// Exception in thread "main" java.lang.ClassCastException:
// java.lang.String cannot be cast to Player
⚠️ Problemi bez Generics-a
  • ❌ Greške u runtime-u - ne u compile-time!
  • ❌ Potreban casting - repetitivan i opasan kod
  • ❌ Nema type safety - možeš staviti bilo što u listu
  • ❌ Teže čitljiv kod - nije jasno što lista sadržava
3

✨ Rješenje: Generics!

❌ BEZ Generics
// Nije type-safe
List players = new ArrayList();
players.add(new Player("Ana"));
players.add("Greška");  // OK 😱

// Casting potreban
Player p = (Player) players.get(0);

// Runtime greška!
Player p2 = (Player) players.get(1);
// 💥 ClassCastException!
✅ SA Generics
// Type-safe!
List<Player> players = new ArrayList<>();
players.add(new Player("Ana"));
players.add("Greška");  // ❌ COMPILE ERROR!

// Bez castinga!
Player p = players.get(0);

// Sigurno - nema runtime greški! ✅
Player p2 = players.get(1);
🎯 Što su Generics?

Generics omogućavaju da definiraš klase i metode s tipovima kao parametrima. Kompajler onda može provjeriti tipove u compile-time, ne u runtime!

🎁 Prednosti Generics-a:

  • ✅ Type safety u compile-time - greške se hvataju odmah
  • ✅ Elimininacija castinga - čitljiviji kod
  • ✅ Reusabilnost koda - jedna klasa/metoda za sve tipove
  • ✅ Bolja dokumentacija - jasno se vidi što metoda prima

🔧 Generičke metode

Sintaksa

public <T> void methodName(T param) {
    // T može biti bilo koji tip!
}

<T> se naziva type parameter i označava da metoda koristi generički tip.

Primjer: Ispis array-a - Generic način
// Jedna metoda za sve tipove! 🎉
public static <T> void printArray(T[] array) {
    for (T element : array) {
        System.out.print(element + " ");
    }
    System.out.println();
}

// Korištenje - radi za sve tipove!
Integer[] numbers = {1, 2, 3, 4, 5};
printArray(numbers);  // Output: 1 2 3 4 5

String[] names = {"Ana", "Marko", "Ivana"};
printArray(names);  // Output: Ana Marko Ivana

Double[] prices = {19.99, 29.99, 39.99};
printArray(prices);  // Output: 19.99 29.99 39.99
❌ Bez Generics - Duplicate kod
// Morali bi napisati 3 metode!
public static void printIntArray(Integer[] arr) {
    for (Integer n : arr) {
        System.out.print(n + " ");
    }
}

public static void printStringArray(String[] arr) {
    for (String s : arr) {
        System.out.print(s + " ");
    }
}

public static void printDoubleArray(Double[] arr) {
    for (Double d : arr) {
        System.out.print(d + " ");
    }
}
// 😫 Copy-paste, teško za održavanje!
✅ S Generics - DRY princip
// Samo jedna metoda! 🎉
public static <T> void printArray(T[] arr) {
    for (T element : arr) {
        System.out.print(element + " ");
    }
}

// Radi za SVE tipove!
// Integer, String, Double,
// Player, User, Post...
// bilo što!

🔤 Konvencije za Type Parameters

Po konvenciji se koriste velika slova za type parameters:

T - Type

Opći tip - najčešće korišteno

public class Box<T> {
    private T item;
}
E - Element

Element u kolekciji

public interface List<E> {
    void add(E element);
}
K, V - Key, Value

Za Map strukture

public interface Map<K, V> {
    V get(K key);
}
N - Number

Za numeričke tipove

public class Calculator<N> {
    N sum(N a, N b);
}
Instagram primjer - Multiple type parameters
// Metoda koja prima DVA generička tipa
public static <K, V> Map<K, V> createMap(K key, V value) {
    Map<K, V> map = new HashMap<>();
    map.put(key, value);
    return map;
}

// Korištenje:
Map<String, Integer> likes = createMap("post_123", 1500);
Map<String, User> followers = createMap("@ana", new User("Ana"));
💡 Dobra praksa
Uvijek koristi konvencionalna imena (T, E, K, V) za type parameters - to će razumjeti svaki Java developer!

📦 Generičke klase

Klasa može imati type parameter koji se koristi kroz cijelu klasu.

Primjer: Box<T> - Generic kontejner
// Generička klasa koja može držati bilo koji tip
public class Box<T> {
    private T content;
    
    public Box(T content) {
        this.content = content;
    }
    
    public T getContent() {
        return content;
    }
    
    public void setContent(T content) {
        this.content = content;
    }
}

// Korištenje:
Box<String> stringBox = new Box<>("Hello Generics!");
String text = stringBox.getContent();  // Bez castinga!

Box<Integer> numberBox = new Box<>(42);
Integer num = numberBox.getContent();  // Type-safe!

Box<User> userBox = new Box<>(new User("Ana"));
User user = userBox.getContent();  // Odlično!
Gaming primjer - Inventory sistem
// Generic inventory za bilo koji tip itema
public class Inventory<T> {
    private List<T> items;
    private int maxSize;
    
    public Inventory(int maxSize) {
        this.items = new ArrayList<>();
        this.maxSize = maxSize;
    }
    
    public boolean addItem(T item) {
        if (items.size() < maxSize) {
            items.add(item);
            return true;
        }
        return false; // Inventory full!
    }
    
    public T getItem(int index) {
        return items.get(index);
    }
    
    public int getSize() {
        return items.size();
    }
}

// Korištenje - različiti inventoryi za različite tipove:
Inventory<Weapon> weapons = new Inventory<>(10);
weapons.addItem(new Weapon("Sword", 50));
weapons.addItem(new Weapon("Bow", 30));

Inventory<Potion> potions = new Inventory<>(20);
potions.addItem(new Potion("Health", 25));
potions.addItem(new Potion("Mana", 15));

// Type-safe - ne možeš staviti Weapon u potions inventory!
// potions.addItem(new Weapon(...)); // ❌ COMPILE ERROR!

🥞 Real-world primjer: Generic Stack

Stack<T> - LIFO struktura podataka
// Generic Stack implementacija
public class Stack<T> {
    private List<T> elements;
    private int maxSize;
    
    public Stack(int maxSize) {
        this.elements = new ArrayList<>();
        this.maxSize = maxSize;
    }
    
    // Push element na vrh stacka
    public void push(T element) {
        if (elements.size() >= maxSize) {
            throw new RuntimeException("Stack overflow! 💥");
        }
        elements.add(element);
    }
    
    // Pop - ukloni i vrati element s vrha
    public T pop() {
        if (isEmpty()) {
            throw new RuntimeException("Stack je prazan!");
        }
        return elements.remove(elements.size() - 1);
    }
    
    // Peek - pogledaj vrh bez uklanjanja
    public T peek() {
        if (isEmpty()) {
            throw new RuntimeException("Stack je prazan!");
        }
        return elements.get(elements.size() - 1);
    }
    
    public boolean isEmpty() {
        return elements.isEmpty();
    }
    
    public int size() {
        return elements.size();
    }
}
Korištenje - Browser History primjer
// Stack za browser back button funkcionalnost
Stack<String> browserHistory = new Stack<>(50);

// Korisnik browsea po stranicama
browserHistory.push("google.com");
browserHistory.push("youtube.com");
browserHistory.push("instagram.com");
browserHistory.push("tiktok.com");

System.out.println("Trenutna stranica: " + browserHistory.peek());
// Output: Trenutna stranica: tiktok.com

// Klikni "Back" button
String previous = browserHistory.pop();
System.out.println("Vraćam se s: " + previous);
// Output: Vraćam se s: tiktok.com

System.out.println("Sada sam na: " + browserHistory.peek());
// Output: Sada sam na: instagram.com
🎯 Zašto je ovo odlično?
Jedna Stack<T> klasa radi za SVE tipove: Stack<String>, Stack<Integer>, Stack<User>, Stack<Post>... Bez Generics-a bi morali pisati zasebnu klasu za svaki tip! 😫

🔒 Bounded Type Parameters

Ponekad želiš ograničiti koje tipove generic može prihvatiti. Koristi extends keyword za upper bound!

Sintaksa - extends keyword
// T mora biti Number ili nešto što nasljeđuje Number
public <T extends Number> void someMethod(T number) {
    // Sada možemo pozvati metode iz Number klase!
    double value = number.doubleValue();
}
❌ Bez bounds - preopćenito
// T može biti BILO ŠTO
public static <T> double sum(List<T> list) {
    double total = 0;
    for (T item : list) {
        // ❌ ERROR: Ne možemo
        // koristiti doubleValue()
        // jer T nije Number!
        total += item.doubleValue();
    }
    return total;
}
✅ S bounds - specifično
// T mora biti Number ili podklasa
public static <T extends Number> 
double sum(List<T> list) {
    double total = 0;
    for (T item : list) {
        // ✅ OK! T extends Number
        // pa ima doubleValue()
        total += item.doubleValue();
    }
    return total;
}
E-commerce primjer - Calculator
// Bounded generic klasa za kalkulacije
public class Calculator<T extends Number> {
    
    public double add(T a, T b) {
        return a.doubleValue() + b.doubleValue();
    }
    
    public double subtract(T a, T b) {
        return a.doubleValue() - b.doubleValue();
    }
    
    public double multiply(T a, T b) {
        return a.doubleValue() * b.doubleValue();
    }
}

// Korištenje:
Calculator<Integer> intCalc = new Calculator<>();
double result1 = intCalc.add(5, 10);  // 15.0

Calculator<Double> doubleCalc = new Calculator<>();
double result2 = doubleCalc.multiply(2.5, 4.0);  // 10.0

// ❌ NE radi - String nije Number!
// Calculator<String> stringCalc = new Calculator<>(); // COMPILE ERROR!

🔗 Multiple Bounds

Tip može imati više ograničenja odjednom!

Sintaksa - & operator
// T mora biti i Number I Comparable
public <T extends Number & Comparable<T>> T max(T a, T b) {
    return a.compareTo(b) > 0 ? a : b;
}
Gaming Leaderboard primjer
// Interface za rangiranje igrača
interface Rankable {
    int getRank();
}

// Klasa mora biti Comparable I implementirati Rankable
public class Leaderboard<T extends Comparable<T> & Rankable> {
    private List<T> players;
    
    public Leaderboard() {
        this.players = new ArrayList<>();
    }
    
    public void addPlayer(T player) {
        players.add(player);
        // Možemo sortirati jer je Comparable
        Collections.sort(players);
    }
    
    public T getTopPlayer() {
        return players.isEmpty() ? null : players.get(0);
    }
    
    public void displayRankings() {
        for (T player : players) {
            // Možemo koristiti getRank() jer implementira Rankable
            System.out.println("Rank " + player.getRank() + ": " + player);
        }
    }
}

// Player klasa koja zadovoljava obje uvjete
class Player implements Comparable<Player>, Rankable {
    private String username;
    private int score;
    private int rank;
    
    @Override
    public int compareTo(Player other) {
        return Integer.compare(other.score, this.score); // Više = bolje
    }
    
    @Override
    public int getRank() {
        return rank;
    }
}
⚠️ Važno pravilo
Ako imaš multiple bounds:
  • ✅ Klasa mora biti prva (ako postoji)
  • ✅ Interface-i dolaze nakon
  • Primjer: <T extends Number & Comparable<T> & Serializable>

🃏 Wildcards - Zamjenski znakovi

Wildcard ? označava "nepoznati tip". Koristi se u situacijama gdje ne znaš ili ne brineš o konkretnom tipu.

Problem: Previše specifična metoda
// Ova metoda radi SAMO za List<Integer>
public static void printList(List<Integer> list) {
    for (Integer i : list) {
        System.out.println(i);
    }
}

// Problem:
List<Integer> ints = List.of(1, 2, 3);
printList(ints);  // ✅ OK

List<String> strings = List.of("a", "b", "c");
printList(strings);  // ❌ ERROR - ne prima String!

List<Double> doubles = List.of(1.5, 2.5);
printList(doubles);  // ❌ ERROR - ne prima Double!
Rješenje: Unbounded wildcard (?)
// ? = bilo koji tip! Univerzalna metoda!
public static void printList(List<?> list) {
    for (Object element : list) {
        System.out.println(element);
    }
}

// Sada radi za SVE tipove! 🎉
List<Integer> ints = List.of(1, 2, 3);
printList(ints);  // ✅ OK

List<String> strings = List.of("a", "b", "c");
printList(strings);  // ✅ OK

List<Double> doubles = List.of(1.5, 2.5);
printList(doubles);  // ✅ OK

List<User> users = getUsers();
printList(users);  // ✅ OK
🤔 Kada koristiti wildcard?
  • ✅ Kada metoda samo čita elemente
  • ✅ Kada ne ovisi o konkretnom tipu
  • ✅ Kada želiš maksimalnu fleksibilnost

⬆️ Upper Bounded Wildcards

<? extends Type> = "bilo koji tip koji JE ili NASLJEĐUJE Type"

Spotify primjer - suma duljina pjesama
// Želimo sumirati duljine pjesama različitih tipova
public static double sumDurations(List<? extends Number> durations) {
    double total = 0;
    for (Number duration : durations) {
        total += duration.doubleValue();
    }
    return total;
}

// Radi za SVE što nasljeđuje Number!
List<Integer> intDurations = List.of(180, 240, 195);  // sekunde
double total1 = sumDurations(intDurations);  // ✅ OK

List<Double> doubleDurations = List.of(3.5, 4.2, 3.8);  // minute
double total2 = sumDurations(doubleDurations);  // ✅ OK

List<Long> longDurations = List.of(180000L, 240000L);  // milisekunde
double total3 = sumDurations(longDurations);  // ✅ OK

// ❌ NE radi - String nije Number!
// List<String> strings = List.of("3:00", "4:00");
// sumDurations(strings); // COMPILE ERROR!
Gaming primjer - hijerarhija karaktera
// Hijerarhija klasa
class Character { 
    protected String name;
    protected int health;
}

class Warrior extends Character { 
    private int armor;
}

class Mage extends Character { 
    private int mana;
}

List<Object> objects = new ArrayList<>();

🎯 PECS Princip

Producer Extends, Consumer Super

Zlatno pravilo za wildcards! ✨

📤 Producer Extends

Kada čitaš/dobavljaš iz kolekcije:

// Čitanje iz liste
public static void processNumbers(
    List<? extends Number> numbers
) {
    for (Number n : numbers) {
        System.out.println(n.doubleValue());
    }
    // ✅ Čitanje - OK!
    // ❌ Pisanje - NE!
    // numbers.add(5); // ERROR
}
📥 Consumer Super

Kada pišeš/dodaješ u kolekciju:

// Pisanje u listu
public static void addIntegers(
    List<? super Integer> list
) {
    list.add(1);
    list.add(2);
    list.add(3);
    // ✅ Pisanje - OK!
    // ⚠️ Čitanje - samo Object
    Object obj = list.get(0);
}
Real-world primjer: Collections.copy()
// Stvarna deklaracija iz Java Collections API
public static <T> void copy(
    List<? super T> dest,     // Consumer (pišemo u dest)
    List<? extends T> src    // Producer (čitamo iz src)
) {
    for (T item : src) {        // Čitanje iz src
        dest.add(item);         // Pisanje u dest
    }
}

// Primjer korištenja:
List<Integer> source = List.of(1, 2, 3);
List<Number> destination = new ArrayList<>();

Collections.copy(destination, source);
// ✅ Radi jer:
// - destination je List<? super Integer> (Number je super od Integer)
// - source je List<? extends Integer> (Integer extends Integer)
💡 Kako zapamtiti PECS?
  • 📤 Producer (izvor) → čitaš podatke → extends
  • 📥 Consumer (odredište) → pišeš podatke → super
  • 🎯 Ako SAMO čitaš i pišeš → ne koristi wildcards, koristi <T>

🔄 Type Erasure - Brisanje tipova

Java Generics koriste "type erasure" - kompajler briše informaciju o tipovima u runtime-u zbog backward kompatibilnosti s Java verzijama prije 5.

📝 Source kod (što pišeš)
// Tvoj kod s Generics
List<String> strings = new ArrayList<>();
strings.add("Java");
strings.add("Generics");

String first = strings.get(0);

// Generic metoda
public <T> void process(T item) {
    System.out.println(item);
}
⚙️ Bytecode (što vidi JVM)
// Nakon type erasure
List strings = new ArrayList();
strings.add("Java");
strings.add("Generics");

String first = (String) strings.get(0);  // Casting!

// T se zamjenjuje s Object
public void process(Object item) {
    System.out.println(item);
}
Type Erasure s bounded types
// Source kod
public class Calculator<T extends Number> {
    private T value;
    
    public double getValue() {
        return value.doubleValue();
    }
}

// Nakon type erasure - T se zamjenjuje s Number (ne s Object!)
public class Calculator {
    private Number value;  // T → Number
    
    public double getValue() {
        return value.doubleValue();
    }
}
⚠️ Posljedice Type Erasure
  • ❌ Ne možeš napraviti new T() - tip ne postoji u runtime-u!
  • ❌ Ne možeš napraviti new T[] - array ne može biti generički
  • ❌ Ne možeš koristiti instanceof T
  • ❌ Statički članovi ne mogu koristiti type parameters
Što NE radi zbog Type Erasure
public class Box<T> {
    // ❌ ERROR - ne možeš instancirati T
    // T item = new T();
    
    // ❌ ERROR - ne možeš napraviti array od T
    // T[] array = new T[10];
    
    // ❌ ERROR - instanceof ne radi s T
    // if (obj instanceof T) { ... }
    
    // ❌ ERROR - statički članovi ne mogu koristiti T
    // private static T staticField;
}

⚠️ Raw Types - Nemoj koristiti!

Raw type = korištenje generičke klase bez type parametara. Postoji samo zbog backward kompatibilnosti s pre-Java 5 kodom.

❌ RAW TYPE - Opasno!
// Bez <> - raw type!
List users = new ArrayList();

// Može primiti SVE - nema type safety!
users.add(new User("Ana"));
users.add("Ovo nije User!");  // 😱 OK u compile-time
users.add(42);  // 😱 OK u compile-time

// Potreban casting - opasno!
User u = (User) users.get(0);  // ✅ OK
User u2 = (User) users.get(1);  // 💥 ClassCastException!

// ⚠️ Compiler warning:
// "List is a raw type. References to 
// generic type List<E> should be parameterized"
✅ GENERICS - Sigurno!
// S <> - type-safe!
List<User> users = new ArrayList<>();

// Samo User objekti!
users.add(new User("Ana"));  // ✅ OK

// Compile-time errors - odlično!
// users.add("String"); // ❌ COMPILE ERROR
// users.add(42); // ❌ COMPILE ERROR

// Bez castinga - sigurno!
User u = users.get(0);  // ✅ Uvijek User
User u2 = users.get(1);  // ✅ Uvijek User

// 🎉 Nema warnings!
TikTok primjer - Raw type problem
// ❌ Loše - Raw type
class VideoQueue {
    private List videos = new ArrayList();  // Raw type!
    
    public void addVideo(Object video) {
        videos.add(video);
    }
    
    public Object nextVideo() {
        return videos.remove(0);
    }
}

VideoQueue queue = new VideoQueue();

✨ Best Practices za Generics

1️⃣ Eliminiraj raw types
// ❌ Loše
List list = new ArrayList();

// ✅ Dobro
List<String> list = new ArrayList<>();
2️⃣ Koristi diamond operator
// ❌ Redundantno
List<String> l = new ArrayList<String>();

// ✅ Kraće (Java 7+)
List<String> l = new ArrayList<>();
3️⃣ Preferiraj liste nad arrayima
// ❌ Array - ne type-safe
Object[] arr = new String[10];
arr[0] = 1;  // Runtime error!

// ✅ Lista - type-safe
List<String> list = new ArrayList<>();
// list.add(1); // Compile error!
4️⃣ Koristi PECS za wildcards
// Producer - extends
void read(List<? extends T> src);

// Consumer - super
void write(List<? super T> dest);
5️⃣ Bounded types kad je potrebno
// Ograniči na Number
public <T extends Number> 
double sum(List<T> list) {
    return list.stream()
        .mapToDouble(Number::doubleValue)
        .sum();
}
6️⃣ Ne koristi instanceof s Generics
// ❌ Ne radi - Type erasure
// if (obj instanceof List<String>)

// ✅ Radi - samo raw type
if (obj instanceof List) {
    List<?> list = (List<?>) obj;
}
💡 Golden Rules
  • ✅ Uvijek koristi Generics umjesto raw types
  • ✅ Koristi wildcards za fleksibilnost
  • ✅ PECS princip za wildcards
  • ✅ Bounded types za ograničenja
  • ❌ Ne pokušavaj "prevariti" type system

🎯 Praktični projekt: Generic Cache

Zadatak: Cache sistem za web aplikaciju

Kreiraj generički cache koji može spremati bilo koji tip podataka s TTL (Time To Live).

CacheEntry - wrapper za podatke
class CacheEntry<T> {
    private final T value;
    private final long expiryTime;
    
    public CacheEntry(T value, long ttlMillis) {
        this.value = value;
        this.expiryTime = System.currentTimeMillis() + ttlMillis;
    }
    
    public boolean isExpired() {
        return System.currentTimeMillis() > expiryTime;
    }
    
    public T getValue() {
        return value;
    }
}
GenericCache - glavna klasa
public class GenericCache<K, V> {
    private final Map<K, CacheEntry<V>> cache;
    private final long defaultTtl;
    
    public GenericCache(long defaultTtlMillis) {
        this.cache = new ConcurrentHashMap<>();
        this.defaultTtl = defaultTtlMillis;
    }
    
    // Dodaj u cache
    public void put(K key, V value) {
        cache.put(key, new CacheEntry<>(value, defaultTtl));
    }
    
    // Dodaj s custom TTL
    public void put(K key, V value, long ttlMillis) {
        cache.put(key, new CacheEntry<>(value, ttlMillis));
    }
    
    // Dohvati iz cache-a
    public Optional<V> get(K key) {
        CacheEntry<V> entry = cache.get(key);
        
        if (entry == null) {
            return Optional.empty();
        }
        
        if (entry.isExpired()) {
            cache.remove(key);  // Očisti expired entry
            return Optional.empty();
        }
        
        return Optional.of(entry.getValue());
    }
    
    // Očisti sve expired entries
    public void cleanup() {
        cache.entrySet()
            .removeIf(entry -> entry.getValue().isExpired());
    }
    
    public int size() {
        return cache.size();
    }
    
    public void clear() {
        cache.clear();
    }
}
Korištenje - Instagram user cache

);

🆕 Generics u Java 25+

Budućnost Generics-a

Project Valhalla i Project Amber donose poboljšanja u Generics!

1. Primitive Type Specialization (budućnost)
// Trenutno - mora boxing
List<Integer> numbers = new ArrayList<>();
numbers.add(5);  // Boxing: int → Integer (overhead!)

// Budućnost - direktno primitive
List<int> numbers = new ArrayList<>();  // Možda uskoro!
numbers.add(5);  // Bez boxinga, brže!
2. Inline Types (Project Valhalla)
// Inline types - value types bez wrappera
inline class Point<T extends Number> {
    private final T x;
    private final T y;
    
    // Brže, manje memorije, bez heap allocation!
}
3. Better type inference
// Trenutno - ponekad treba eksplicitno
List<String> list = Collections.<String>emptyList();

// Java 25+ - još bolji inference
var list = List.of("a", "b", "c");  // List<String> inferred
var map = Map.of("key", 1, "key2", 2);  // Map<String, Integer>
